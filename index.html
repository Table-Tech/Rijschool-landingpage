Rijschool Planning Systeem — Onepager
Versie 1.0 • 25 maart 2026

Volledig overzicht van het multi-tenant rijschoolplatform

Het Rijschool Planning Systeem is een volwassen SaaS-oplossing voor rijscholen met duidelijke rolscheiding, centrale planning, leerlingbeheer, instructeurbeheer, betalingen, communicatie, auditlogging en platformbeheer.

Architectuur Monorepo (Frontend + Backend + Super Admin)
Tenant model Strikte data-isolatie per rijschool
Kernrollen Super Admin, Eigenaar, Instructeur, Leerling
Status Production-ready* (met deployment-checks)

1. Systeem in 1 minuut

Wat het systeem doet

  • Plannen, wijzigen en beheren van rijlessen in één omgeving.
  • Complete leerling- en instructeuradministratie met voortgang.
  • Betalingen, pakketten en openstaande posten centraal beheerd.
  • Emailflows, reminders, audits en exports voor compliance.
  • Super-admin laag voor tenants, features en platformcontrole.

Voor wie

Super Admin

Platformbeheer, tenants, feature flags, lifecycle tooling.

Eigenaar

Hoogste rol binnen rijschool: planning, team, finance, audit.

Instructeur

Rooster, leerlingen, lesuitvoering en voorstelacties.

Leerling

Eigen lessen/profiel en relevante statusinformatie.

2. Architectuur & stack

Frontend

Next.js 15 (App Router), React 19, TypeScript, Tailwind, shadcn/ui.

Backend

Node.js + Express + PostgreSQL, JWT auth met refresh token flow.

Super Admin

React + Vite portal voor platformniveau beheer en audit.

Integraties

  • Resend (email)
  • Mollie (betalingen + webhooks)
  • Cron jobs (reminders/installments)
  • Pino logging en optioneel Sentry

Data & security fundament

Multi-tenant isolatie via rijschool_id, parameterized SQL, inputvalidatie en endpoint-specifieke rate limits.

4. Eigenaar-rol (centrale beheerrol)

Belangrijkste rechten

  • Alle instructeurs en leerlingen binnen eigen rijschool beheren.
  • Centrale planning overzien, bijsturen en goedkeuren.
  • Financieel beheer, pakketbeheer en herinneringen.
  • Instellingen, emailbeheer en audit-inzichten.

Flexibel permissiemodel

De eigenaar kan optioneel ook instructeurrechten dragen via has_instructor_role true/false.

9. Recente oplossingen (25 maart 2026)

Operationele fixes

  • ENV opgeschoond: duplicaten/verkeerde placeholders verwijderd.
  • Route-middleware gestandaardiseerd naar verifyToken.
  • Super-admin build gefixt via api alias export.
  • Dev DB SSL chain fout opgelost met lokale override.
  • CORS allowlist uitgebreid voor lokale origins.
  • EADDRINUSE opgelost via 1 actieve backend instance.

Bekende dev-waarschuwingen (geen blockers)

  • Sentry DSN ontbreekt → monitoring niet actief.
  • Resend API key ontbreekt → email reminders uit.
  • SSL validatie waarschuwing door dev-override.

10. Runbook & verificatie

Lokale startvolgorde

  1. Dependencies installeren (root + backend + super-admin).
  2. ENV-bestanden controleren.
  3. Backend starten op poort 5000.
  4. Frontend starten op poort 3000.
  5. Super-admin starten op poort 5173 (optioneel).

Als iets faalt

  • EADDRINUSE → proces op poort stoppen.
  • CORS error → ALLOWED_ORIGINS controleren.
  • DB SSL error → dev SSL reject instelling checken.
  • Auth errors → JWT_SECRET lengte/waarde controleren.

Regressiechecks

  • Login per rol: eigenaar, instructeur, leerling.
  • CORS preflight op auth endpoint.
  • Belangrijkste dashboard/data endpoints.
  • Lesvoorstel-workflow end-to-end.
  • Reminder flow.
  • Basis payment flow.
